# Stakeholder Objection Bank

Pre-built objection tables for the three primary gatekeepers. Tailor these to the user's specific situation, adding or removing rows as needed. The "concession/guardrail" column is critical because it shows you have thought about their concerns before they raise them.

---

## Procurement / Finance

| Objection | What they are really protecting | Best response | Proof to bring | Concession / guardrail to offer |
|-----------|-------------------------------|---------------|----------------|-------------------------------|
| "What's the ROI? Why can't you do this with existing tools or headcount?" | Budget discipline, avoiding redundant spend | Quantify the status quo cost: hours spent on manual processes × fully loaded rate, plus rework/error cost. Show that existing tools lack the capability (e.g., no pay equity modelling in current HRIS). | Time-motion data from the team, error/rework logs, vendor comparison showing feature gaps in current stack. | Offer a time-boxed pilot (e.g., 90 days) with a defined success metric. Agree to a go/no-go review before any enterprise commitment. |
| "We already have too many vendors. Why add another?" | Vendor sprawl, integration burden, contract management load | Acknowledge the concern, then show this tool replaces or consolidates existing manual workarounds (spreadsheets, consultancy engagements, ad-hoc data requests). | A list of current workarounds and their costs (consultancy fees, analyst time, manual data pulls). | Agree to sunset one existing tool or process if the pilot succeeds. Propose integration via API rather than standalone. |
| "The pricing doesn't look competitive." | Value for money, negotiation leverage | Benchmark against alternatives (build vs. buy analysis, consultancy cost for equivalent output). Show cost per use case, not just licence fee. | Vendor pricing breakdown, competitor quotes or published pricing, internal build estimate from IT. | Negotiate volume discounts, phased rollout pricing, or a success-based pricing model for the pilot. |
| "What happens when the contract ends? Is there data lock-in?" | Continuity risk, switching costs | Confirm data portability: can you export all data in standard formats? What is the offboarding process? | Vendor's data portability clause, export functionality documentation, SLA for data return on termination. | Require a contractual data portability and deletion clause. Request an offboarding runbook from the vendor. |
| "You haven't accounted for the real costs. What about data prep, integration, change management?" | Financial realism, protecting against budget overruns | Agree. Include these costs explicitly in the business case. This is not a weakness; it is a strength. A case that pre-empts hidden costs is far more credible than one that ignores them. Use the hidden costs section in `references/roi-model.md` to structure the estimate. | Itemised cost estimate covering: data preparation/cleaning, systems integration, training/upskilling, ongoing monitoring, and a contingency buffer (15-20%). | Propose a "General Experimentation Fund" approach: a small, ring-fenced budget (£5k-15k) for a 90-day proof of concept that tests costs and benefits before any enterprise commitment. |
| "Who owns the intellectual property if the AI generates a novel compensation structure or model?" | IP risk, vendor overreach | Confirm that the contract defines IP ownership of outputs explicitly. Outputs generated from your data should belong to you. Check whether the vendor claims any rights to derivative works or model improvements. | Vendor contract IP clause, definitions of "Input," "Output," "Customer Data," and "Trained Customer Model." | Require a contractual clause stating that all outputs generated from your data are your exclusive property. Ensure the vendor cannot use your outputs to improve their product for competitors. |

## IT / Security

| Objection | What they are really protecting | Best response | Proof to bring | Concession / guardrail to offer |
|-----------|-------------------------------|---------------|----------------|-------------------------------|
| "Where does pay/PII data go, who can access it, and is it retained or used to train models?" | Data sovereignty, breach risk, regulatory compliance | Map the data flow end-to-end: what data leaves your environment, where it is processed, who can access it, how long it is retained, and whether it is used for model training. | Vendor's data processing architecture diagram, data residency confirmation, model training opt-out documentation, SOC 2 Type II report. | Require contractual data processing agreement (DPA) with explicit no-training clause. Offer to run a data classification exercise before any data leaves the environment. |
| "Does this meet our security standards? (SOC 2, ISO 27001, pen testing)" | Compliance with internal security policy | Confirm vendor certifications and ask what gap remains. Offer to facilitate a vendor security questionnaire. | Vendor's SOC 2 Type II report, ISO 27001 certificate, most recent pen test summary, security questionnaire responses. | Agree that IT runs their standard vendor security assessment before any data is connected. Offer to co-own the review. |
| "How does this integrate with our existing systems? What is the attack surface?" | System stability, integration risk, lateral movement | Show the integration is via secure API (OAuth 2.0, scoped tokens), not direct database access. Confirm the tool operates in a sandboxed environment. | API documentation, architecture diagram showing integration points, vendor's shared responsibility model. | Agree to a staged integration: read-only access first, then write access only after security review. Limit initial scope to non-production data or anonymised datasets. |
| "What happens if there is a breach or outage?" | Incident response readiness, SLA enforcement | Confirm the vendor's incident response plan, notification SLA, and your internal escalation path. | Vendor's incident response policy, SLA terms, breach notification timeline (e.g., 72 hours per GDPR). | Require a contractual incident notification clause aligned to your internal policy. Agree to a joint tabletop exercise before go-live. |
| "Can employees access data they shouldn't?" | Least-privilege access, insider threat | Confirm RBAC (role-based access control), audit logging, and that access is limited to named users with business justification. | Vendor's access control documentation, audit log samples, SSO/SAML integration capability. | Agree to quarterly access reviews. Implement SSO integration so access is governed by your identity provider. |
| "People are already using consumer AI tools. Why do we need an enterprise licence when the free version works?" | Shadow AI risk, governance gap, data leakage | This is actually your strongest argument, not theirs. Reframe: "Exactly. Our people are already sending sensitive pay data and employee PII to unencrypted, unmonitored consumer platforms. An enterprise-licensed tool brings this under your security perimeter, with encryption, access controls, audit trails, and a no-training clause. The question is not whether we use AI. It is whether we govern it." | Evidence of Shadow AI usage (anonymous survey, anecdotal reports, IT logs showing access to consumer AI platforms). Research showing ~8.5% of employee prompts to public AI tools contain sensitive corporate data. | Propose that the enterprise tool rollout includes a clear policy replacing unsanctioned consumer AI use. Position IT as the hero who brought AI under governance, not the blocker who failed to prevent data leakage. |

## Legal / Privacy / Employee Relations

| Objection | What they are really protecting | Best response | Proof to bring | Concession / guardrail to offer |
|-----------|-------------------------------|---------------|----------------|-------------------------------|
| "Could this create discrimination risk or automated decision-making claims? Can managers explain outcomes?" | Regulatory compliance (GDPR Art. 22, local employment law), litigation risk | Clarify that the tool is decision-support, not decision-making. Managers review, can override, and must document rationale for any employment outcome. Outputs include an explanation layer. | Vendor's explainability documentation, sample output showing rationale trail, your proposed human-in-the-loop policy. | Commit to a DPIA before go-live. Draft a Responsible AI Usage Policy that mandates human review for any output that informs pay, promotion, or termination. Offer Legal a seat on the governance committee. |
| "What is the legal basis for processing this data?" | GDPR/local privacy law compliance, data subject rights | Identify the legal basis (likely legitimate interest or performance of contract for employment data). Confirm data minimisation: only the data strictly necessary is processed. | Data protection impact assessment (DPIA), record of processing activities (ROPA) entry, vendor's DPA. | Complete the DPIA before pilot launch. Add the processing activity to the ROPA. Provide employee transparency notice. |
| "What if an employee challenges an AI-influenced decision?" | Defensibility in tribunal/litigation, duty of care | Show the audit trail: every AI output is logged, the human decision-maker is recorded, and the rationale is documented. The AI recommendation can be separated from the human decision. | Sample audit trail, proposed record-retention policy, vendor's logging and export capabilities. | Agree to a record-retention period aligned to your employment litigation limitation period. Implement a "right to human review" process for employees. |
| "Who is liable if the AI gets it wrong?" | Contractual risk allocation, indemnification | Clarify that liability sits with the organisation (you), not the tool, because the tool provides recommendations, not decisions. Confirm the vendor's indemnification and limitation of liability clauses. | Vendor contract terms (indemnity, LoL), your proposed governance framework showing human accountability at each step. | Negotiate appropriate vendor indemnification for data breaches. Ensure your AI usage policy makes clear that the human decision-maker is accountable for outcomes. |
| "We don't have an AI policy yet. Shouldn't we wait?" | Governance gap, fear of setting precedent | Argue that a controlled pilot is the best way to develop a fit-for-purpose policy, rather than writing policy in the abstract. Offer to co-author the policy with Legal. | Examples of AI policies from peer organisations (anonymised or public), your proposed policy outline. | Propose that the pilot is conditional on an AI usage policy being in place before go-live. Offer to draft v1 and have Legal review. This turns a blocker into a deliverable. |
| "This could trigger Works Council / union consultation requirements." | Labour relations obligations, consultation timelines | Acknowledge and plan for it. Identify whether the tool falls within consultation scope (usually: if it monitors, profiles, or makes decisions about employees). Map the consultation timeline. | Legal analysis of consultation triggers, proposed timeline, precedent from similar tool introductions. | Build the consultation into the project plan from day one. Offer transparency: share the tool's purpose, scope, and safeguards with employee representatives early. |
| "The data isn't truly anonymised. AI can re-identify individuals from pseudonymised datasets." | Re-identification risk, regulatory exposure | They are right, and you should not argue otherwise. Modern AI can triangulate disparate data points to re-identify individuals, so Legal teams increasingly treat all HR data as pseudonymised (still personal data under GDPR), not anonymised. Build this assumption into your governance framework from the start. | Research on re-identification risk, your DPIA explicitly treating data as pseudonymised, data minimisation plan showing only strictly necessary fields are processed. | Agree to treat all data as personal data regardless of any anonymisation steps. Implement data minimisation: only input the fields the tool actually needs, not the full employee record. Propose regular re-identification risk audits. |

## Bonus: HR Leadership / CHRO

| Objection | What they are really protecting | Best response | Proof to bring | Concession / guardrail to offer |
|-----------|-------------------------------|---------------|----------------|-------------------------------|
| "Is this a priority right now?" | Resource allocation, competing initiatives, change fatigue | Connect to an existing strategic priority (e.g., pay transparency legislation, M&A harmonisation, benefits renewal). Show that the tool accelerates something already on the agenda. | Alignment to current HR strategy document, compliance deadline, or board-level priority. | Propose a lightweight pilot that does not require additional headcount or significant change management. |
| "Will this replace people on the team?" | Team morale, trust, change management | Frame as augmentation: the tool handles repetitive data work so the team can focus on advisory and strategic work. Show how roles evolve, not disappear. | Task analysis showing % of time spent on automatable vs. advisory work, case studies from similar teams. | Commit to a communication plan for the team. Involve team members in the pilot as subject-matter experts. |
